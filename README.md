# 飲食店向け勤怠管理システム

## 1. 目的

本システムは、小規模飲食店における勤怠管理業務の効率化を目的として開発する。

紙のタイムカードやExcelによる手動集計では、以下の課題が発生している。

- 集計ミスの発生
- 月末の業務負荷増大
- 打刻修正履歴の不透明さ
- シフトと実績の差異把握が困難

これらの課題を解決するため、Webアプリケーションとして勤怠管理システムを構築する。

---

## 2. 想定利用環境（背景設定）

### 想定クライアント

- 地方の個人経営飲食店
- 従業員数：約10名（アルバイト中心）
- 店長1名が勤怠管理を担当
- 現在は紙タイムカードで管理

### 現状業務フロー

1. 出勤時に紙へ打刻
2. 退勤時に手書き記録
3. 月末に店長が手動集計
4. 給与計算へ反映

### 現状の問題点

- 手計算による集計ミス
- 修正時の証跡が残らない
- 月末に業務が集中する
- 過去データの検索性が低い

---

## 3. 要件分析

### 業務上必要な機能要素

- 従業員ごとの出退勤記録管理
- 勤務時間の自動計算
- 休憩時間の管理
- 打刻修正申請および承認フロー
- 月次集計機能
- データ出力機能（CSV）
- 日をまたぐ勤務（深夜帯シフト）への対応

### 利用者区分

#### 従業員（アルバイト）

- 出勤・退勤打刻
- 自身の勤怠確認
- 打刻修正申請
- 自身の修正申請履歴の確認

#### 管理者（店長）

- 全従業員の勤怠一覧確認
- 修正申請の承認／却下
- 月次集計確認
- CSV出力
- 従業員の登録・管理・退職処理

---

## 4. 要件定義

### 4.1 機能要件

1. 認証機能  
   - メールアドレス・パスワードによるログイン  
   - ロール（従業員／管理者）によるアクセス制御  
   - ログイン後はロールに応じて異なる画面へリダイレクト（従業員：打刻ダッシュボード、管理者：管理者ダッシュボード）  
   - パスワードは各従業員が**個人スマートフォンから初回設定**を行う
   - 管理者はアカウント（メールアドレス／従業員番号）のみ発行し、パスワードは保持しない
   - ログインは**店舗端末からのみ可能**とする（私物端末からのログインは禁止）
   - パスワード再設定は個人スマートフォンから実施可能とする
   - パスワード未設定アカウントはログイン不可とし、メールリンクからの設定を促す
   - 一定時間操作がない場合は自動ログアウトを行う
   - 打刻完了後は自動ログアウトを行う
    
2. 打刻機能  
   - 出勤打刻  
   - 退勤打刻  
   - 休憩開始／終了記録（1日に複数回の分割休憩に対応）  
   - 日をまたぐ勤務（深夜帯シフト）を許容する  
   - 勤務日は出勤打刻日を基準とする  

3. 勤務時間計算機能  
   - 複数回の休憩時間をすべて合算し、実働時間を自動算出  
   - 日別詳細画面で各休憩の開始・終了を時系列で確認可能  
   - 月別の合計実働時間表示  
   - 日をまたぐ勤務でも正確な実働時間を算出する  
   - 月別集計・CSV出力は出勤日基準で行う  

4. 修正申請機能  
   - 従業員が打刻修正を申請可能  
   - 従業員が自分の申請履歴（審査中／承認済／却下済）を確認可能  
   - 管理者が承認／却下可能  
   - 承認履歴の保存  

5. 月次集計機能  
   - 指定月の勤務時間一覧表示  
   - 従業員別集計表示  

6. CSV出力機能  
   - 月次勤怠データをCSV形式で出力  
   - 出力項目：従業員番号・氏名・日付・出勤時刻・退勤時刻・休憩時間合計・実働時間  
   - 休憩時間合計は当日の全分割休憩を合算した値を出力  
   - 従業員番号は給与計算ソフトの従業員コードと紐づけられる

7. 従業員管理機能（管理者専用）  
   - 従業員の登録（氏名・メールアドレス・従業員番号）。登録後、パスワード設定用リンクをメール送信する（管理者はパスワードを保持しない）  
   - 有効な従業員の一覧表示  
   - 退職処理（論理削除）。物理削除は行わず過去の勤怠データを保持する  
   - 氏名・プロフィール等の編集は管理者のみ可能とし、従業員自身による変更は行えない  
   - 従業員データは物理削除を行わず、論理削除で管理する
---

## 4.3 セキュリティ要件（多層防御）

### アクセス制御
- ログインは店舗端末からのみ可能とする  
- 店舗外からのアクセスを防ぐため、IPアドレスによるアクセス制限を行う  
- 許可するIPアドレスは店舗ネットワーク（Wi-Fi）のグローバルIPに限定する  

### パスワード設定・再設定
- パスワード設定および再設定は、個人スマートフォンから実施可能とする  
- 管理者は従業員作成時にメールアドレスを登録し、トークン付きのパスワード設定用リンクを送信する  

### トークン制御
- ワンタイムトークンによる認証  
- 有効期限を設定（例：24時間以内）  
- 使用後は無効化  

### アクセス制御の例外
- 以下のエンドポイントはIP制限対象外とする  
  - パスワード設定画面  
  - パスワード再設定画面  

### ログイン制御
- 通常ログイン画面はIP制限対象とし、店舗外からのログインを禁止する  

### データ保護
- パスワードはハッシュ化して保存する  

※ 本仕様は利便性とセキュリティのバランスを考慮した多層防御構成とする

---

### 4.3 非機能要件

- パスワードはハッシュ化して保存する  
- 同時アクセス20名までを想定する  
- 月末集計時でも処理遅延が発生しない設計とする  
- データの永続化およびバックアップを前提とする  
- スマートフォンからの利用を考慮したUI設計とする  
- サーバおよびアプリケーションのタイムゾーンは **JST（Asia/Tokyo）固定**とする。日をまたぐ判定・月集計の日付境界は全てJSTで評価する  


5. ■ PHP 8.5

最新の安定版であり、パフォーマンス改善や型安全性の向上など、継続的な言語進化の恩恵を受けられるため採用しました。 Laravel 12との互換性も確保されており、今後の長期運用を見据えたバージョン選定としています。

■ Laravel 12

現行の公式サポート対象バージョンであり、軽量かつシンプルな構成を採用している点を評価しました。 最新のベストプラクティスに基づいた開発を行うことで、保守性・拡張性の高いアプリケーション構築を目指しています。

■ MySQL 8.4

最新の長期サポート（LTS）版であり、トランザクション管理やインデックス機能などが成熟しているRDBMSです。 実務利用実績が豊富であり、安定性と信頼性を重視して採用しました。

■ Docker（Laravel Sail）

開発環境の再現性を確保し、チーム開発を想定したコンテナベースの構成を採用しました。 ローカル環境差異を排除し、本番環境に近い構成で開発を行うことを目的としています。

■ PHPUnit

Laravel標準のテストフレームワークを使用し、主要機能の動作保証と回帰防止を目的としています。 継続的なリファクタリングを可能にするため、テスト駆動の開発を意識しています。

■ GitHub Actions

プッシュ時に自動でテストを実行するCI環境を構築しています。 コードの品質担保と、継続的インテグレーション（CI）を実現するために導入しています。
